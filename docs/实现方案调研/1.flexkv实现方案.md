## 1. 先说结论

FlexKV 不是一个“把 Attention 算子改快”的项目。

它做的事情更像是：

- 把大模型推理过程里产生的 KV Cache 当成一种可以检索、缓存、搬运、复用的数据资产。
- 尽量避免同一段前缀被重复计算。
- 当 GPU 显存不够时，不急着丢掉 KV，而是把它放到 CPU、SSD，甚至其他机器上。
- 当下一次又遇到相同前缀时，优先把 KV 取回来，而不是重新跑一遍 Prefill。

一句话概括：

> FlexKV 的核心思想是“用前缀匹配找到已经算过的 KV，再用分层缓存和异步传输把它快速拿回来”。


## 2. 它到底解决了什么问题

### 问题 1：同样的前缀总被重复算

很多在线推理场景里，前缀重复非常常见：

- 多轮对话里，系统提示词每次都一样。
- RAG 里，模板、指令、固定上下文常常重复。
- Agent 工作流里，同一种工具调用框架会反复出现。

如果没有 KV 复用，同样的前缀每次都要重新 Prefill，GPU 算力会被大量浪费。

### 问题 2：GPU 显存太贵，也太小

KV Cache 很占显存，尤其是长上下文、多并发时更明显。

传统做法常常是：

- GPU 放不下了，就把老 KV 丢掉。
- 以后再碰到相同前缀，再重新计算。

这等于把“显存不够”直接变成“重复算力开销”。

### 问题 3：命中了外部缓存，不代表就一定快

即便 KV 已经在 CPU 或 SSD 上，如果取回路径是串行的、阻塞的，收益也会被传输开销吃掉。

### 问题 4：多机部署时，不同机器会重复做同一份 Prefill

如果集群里多个节点都可能收到相似请求，那么仅仅在单机内缓存是不够的。
还需要跨节点复用 KV。

如果只有单机缓存，那么只能做到：

  - 机器 A 复用自己以前算过的 KV
  - 机器 B 复用自己以前算过的 KV

  但做不到：

  - 机器 B 直接使用机器 A 已经算过的 KV


## 3. 它用了哪些核心算法和机制

FlexKV 不是靠单一算法提速，而是靠几种机制配合。

### 3.1 块级前缀哈希

这是它最核心的第一步。

FlexKV 不按 token 一个个管理 KV，而是把 token 按固定大小切成 block。
比如每 16 个 token 组成一个 block。

然后它不是给“单个 block 内容”做哈希，而是给“到当前 block 为止的整个前缀”做哈希。

也就是说：

- 第 1 个 block 的 hash：表示前 1 个 block 的前缀
- 第 2 个 block 的 hash：表示前 2 个 block 的前缀
- 第 3 个 block 的 hash：表示前 3 个 block 的前缀

这样做的好处是：

- 两个请求只要前缀相同，它们前面若干个 block 的 hash 序列就完全一样。
- 一旦中途分叉，后面的 hash 就会不同。

这非常适合做“最长公共前缀匹配”。

### 3.2 RadixTree 最长前缀匹配

有了前缀 hash 序列之后，FlexKV 用 RadixTree 来做索引。

你可以把它理解成一棵“前缀树”：

- 树上的路径代表一段 token 前缀。
- 每个节点记录这段前缀对应的物理 block 在哪里。

当新请求到来时，FlexKV 会沿着这棵树往下走，找到当前请求能匹配到的最长前缀。

这样它就知道：

- 前面多少个 block 已经有 KV 了
- 这些 KV 存在 CPU、SSD 还是远端
- 从哪个物理 block 可以把它们取回来

#### 和vllm索引的比较
 ## 1. vLLM 的索引结构更像什么

  更像一个 “标签 -> 箱子” 的表。

  意思是：

  - 每个 block 都有一个 hash 标签
  - 系统维护一张表
  - 这张表告诉你：这个 hash 对应哪个 block

  你可以把它想成：

  标签A -> 箱子5
  标签B -> 箱子12
  标签C -> 箱子20

  请求来了以后，vLLM 会做什么？

  - 先把请求切成 block
  - 再算出每个 block 的 hash
  - 然后从前往后查：
      - 第一个 hash 在不在表里
      - 第二个在不在
      - 第三个在不在
  - 一旦某个不在，就停

  所以它更像是在做：

  > “这个 block 的标签我认不认识？”

  它主要关心的是：

  - 这个 block 有没有现成缓存
  - 能连续命中多少个 block

  这对于 GPU 内部前缀缓存 已经够用了。


  ## 2. FlexKV 的索引结构为什么不一样

  因为 FlexKV 不只是问：

  - “这个 block 有没有命中？”

  它还要继续问：

  - 这段前缀命中到了哪里？
  - 命中的这段是不是已经真的可用？
  - 这段是在 CPU 里，SSD 里，还是远端？
  - 如果我要把后面新内容接上，应该从哪接？
  - 如果这段正在传输，还能不能拿来用？
  - 如果空间不够，哪些节点能被驱逐？

  所以它面对的已经不是一个简单的“查标签”问题了。

  它更像是在维护一棵 “前缀关系树”。

  ———

  ## 3. 为什么树比表更适合 FlexKV

  这里给你举个非常直白的例子。

  假设有三段请求：

  A: 你是一个助手 / 请总结文章 / 输出三点
  B: 你是一个助手 / 请总结文章 / 输出五点
  C: 你是一个助手 / 请翻译文章 / 输出中文

  它们的关系不是彼此独立的，而是有公共前缀：

  你是一个助手
      ├── 请总结文章
      │      ├── 输出三点
      │      └── 输出五点
      └── 请翻译文章
             └── 输出中文

  这时候如果你只是用“标签 -> block”表去查，当然也能查一部分。
  但 FlexKV 更关心的是这整段前缀关系：

  - 哪一段是公共祖先
  - 命中最长到哪
  - 分叉点在哪里
  - 新节点应该挂在哪棵子树下

  所以树结构就天然更适合它。

  ———

  ## 4. FlexKV 的树多管了哪些事情

  vLLM 更像是：

  - “有没有这块缓存”

  FlexKV 更像是：

  - “这一整段前缀现在走到了树上的哪个节点”
  - “这个节点是不是 ready”
  - “这个节点是不是正在用，不能驱逐”
  - “这段前缀后面如果继续新增，应该从哪个节点往下接”
  - “这个节点对应的物理数据到底在哪一层”

  所以它不是单纯的“查 hash”，而是：

  > 一边查前缀，一边管理这段前缀当前的生命周期和位置。

  这就是为什么它要用 RadixTree，而不只是简单哈希表。

### 3.3 多级缓存

FlexKV 的存储层次通常是：

- GPU：正在参与计算的 KV
- CPU：第一层外部缓存
- SSD：第二层缓存
- Remote：第三层远端共享缓存

它的目标不是“所有 KV 都塞回 GPU”，而是让 GPU 只保留最需要的部分，把更大的容量放到更便宜的存储上。

### 3.4 异步传输图
如果一份 KV 不在 GPU，而是在 CPU、SSD 或别的机器上，那就得把它搬回来。
  > 但“搬回来”不是一步完成的，往往要经过好几站。

FlexKV 不会把一次读取写成“先从 SSD 读，再拷到 CPU，再拷到 GPU”的同步大串行操作。

 FlexKV 的想法是：

  > 不要把“取回 KV”写成一大坨顺序代码，
  > 而是先把它拆成很多小操作，再告诉系统这些操作之间谁依赖谁。

  这个“很多小操作”就叫：

  - TransferOp
operation操作

  这些小操作之间的依赖关系连起来，就成了一张图：

  - 哪些步骤能先做
  - 哪些步骤必须等前面做完
  - 哪些步骤可以并行

  所以“异步传输图”你可以直接理解成：

  > 一张“搬运计划表”。


常见的传输类型包括：

- `D2H`：GPU -> CPU
- `H2D`：CPU -> GPU
- `DISK2H`：SSD -> CPU
- `H2DISK`：CPU -> SSD
- `REMOTE2H`：远端存储 -> CPU
- `DISK2D`：SSD -> GPU，走 GDS
- `PEERH2H`：远端 CPU -> 本地 CPU
- `PEERSSD2H`：远端 SSD -> 本地 CPU

然后 TransferEngine 会异步调度这些 op，并尽量让传输和计算重叠。

### 3.5 驱逐算法

缓存不可能无限大，所以 FlexKV 还实现了驱逐策略。

支持：

- LRU
- LFU
- FIFO
- MRU
- FILO

LRU = Least Recently Used
  意思是：

  > 最久没有被使用的数据，优先删掉

 LFU = Least Frequently Used
  意思是：

  > 命中次数最少的数据，优先删掉
  
  MRU = Most Recently Used
  意思是：

  > 最近刚刚被用过的数据，优先删掉
  
  普通 LRU 有个弱点：

  > 它只看“最近有没有用过”，不太看“总共用了多少次”。 

FlexKV 又加了一个补丁机制：

  - hit_reward_seconds

每次命中，系统都会给这个节点额外加一点“保护时间”。
  也就是说：
  - 普通 LRU 只看“最近访问时间”
  - FlexKV 的这个机制会让“命中很多次的节点”在时间上被额外保护

所以，
- 命中越频繁，节点的“有效存活时间”越长
- 这样热点 KV 更不容易被过早淘汰

### 3.6 分布式索引快照 + Lease

跨节点场景下，FlexKV 不是每次查询都去 Redis 实时查全局索引。

它的做法是：

- 每个节点维护本地权威索引
- 同时从 Redis 拉取其他节点发布的元数据，构建一个远端索引快照

这样查询时只查本地内存里的快照，不走中心化同步查表。

另外，它还引入 Lease / TTL 机制，避免你查到了远端 KV，但对方已经删了或者正在回收。


### 3.7 冷热数据怎么判断和处理

先说结论：

> FlexKV 靠“分层放置 + 驱逐策略 + 命中保护”来近似判断冷热。

可以把它理解成下面几层意思。

#### 第一层：默认位置本身就带有冷热含义

在 FlexKV 里，不同存储层天然就代表不同温度：

- GPU：最热，最近最可能立即参与计算
- CPU：次热，虽然不在 GPU，但仍然比较值得快速取回
- SSD：更冷，容量更大，但取回更慢
- Remote：更偏共享备份或跨节点复用层，通常比本地层更冷

#### 第二层：最近访问过的更热

FlexKV 支持 `LRU` 这类策略，核心思想就是：

- 最近刚被访问过的 KV 更可能很快再次被访问
- 很久没用过的 KV 更可能被下沉或驱逐

所以如果一个前缀刚刚命中过，它在系统眼里就会变得更“热”。

#### 第三层：命中频率高的更热

除了“最近用没用过”，FlexKV 还支持 `LFU` 这类思路，也就是：

- 被命中过很多次的 KV，比只偶尔命中过一次的 KV 更值得保留

另外它还有一个 `hit_reward_seconds` 机制，可以把它理解成：

- 每次命中，系统都给这个节点额外加一层保护时间
- 命中越频繁，就越不容易被过早驱逐

这相当于在“最近访问”之外，又额外考虑了“访问频率”。

#### 第四层：正在被使用的数据，不算冷数据

FlexKV 里有 `lock` 和 `ready` 这样的状态。

这意味着：

- 正在被使用的节点不能随便驱逐
- 还没传输完成、还没 ready 的节点也不能当作普通冷数据处理

所以系统判断冷热时，不只是看“值不值得保留”，还要看“当前能不能动”。

也就是说：

- 真正正在参与请求的数据，一定优先被保护
- 不是说它理论上偏冷，就能马上驱逐

#### 第五层：冷数据不是立刻消失，而是先往下层走

FlexKV 的思路更像是：

- 最热的放 GPU
- 没那么热的放 CPU
- 更冷的往 SSD 放
- 更远的还能在 Remote 层继续保存

所以它对冷数据的处理不是单纯“删除”，而是优先“下沉”。

这意味着冷热管理在 FlexKV 里不只是：

- 留下谁
- 删除谁

还包括：

- 谁应该继续留在上层
- 谁应该降到下层
- 谁才真的应该被彻底驱逐

#### 第六层：它更像工程化冷热管理，不是复杂预测模型

所以如果你问：FlexKV 有没有像推荐系统那样专门训练一个冷热预测器？

更准确的答案是：没有，它主要不是那条路线。

它更像一种工程化方案：

- 通过分层位置表达冷热
- 通过 `LRU/LFU/FIFO/MRU/FILO` 决定驱逐顺序
- 通过 `hit_reward_seconds` 保护热点
- 通过 `lock/ready` 保护正在使用的数据

所以这套系统的冷热判断重点不是“预测未来”，而是：

> 根据最近访问、访问频率、当前层级和当前状态，尽量把更有价值的 KV 留在更容易取回的位置。



## 4. 一个简单例子：为什么要用“前缀哈希 + 前缀树”

假设每 4 个 token 一块。

请求 A：

```text
[系统提示][用户问题1][用户问题2][用户问题3]
```

请求 B：

```text
[系统提示][用户问题1][用户问题2][另一个问题]
```

如果按 block 切分，A 和 B 的前 3 个 block 是一样的，只是最后一个 block 不一样。

FlexKV 就能得出结论：

- 前 3 个 block 对应的 KV 可以复用
- 只需要重新计算最后那个分叉后的 block

如果没有这种机制，B 通常要把整段前缀重新 Prefill 一遍。


## 5. 它的读取流程是怎样的

这里说的是“已经算过的 KV，下一次怎么拿回来”。

### 第一步：把请求整理成 block

请求来的时候，FlexKV 会：

- 对齐到完整 block
- 生成 block 级前缀 hash
- 根据 `slot_mapping` 知道 GPU 侧要放到哪些物理 block

### 第二步：查索引

它会在不同层上做前缀匹配：

- CPU 索引
- SSD 索引
- Remote 索引

从而得到“最长命中前缀有多少块”。

### 第三步：把命中的部分分段处理

可以简单理解成三段：

- 一段已经在 CPU 上，直接可拷
- 一段只在 SSD 上，需要先读到 CPU
- 一段只在 Remote 上，需要先从远端拿回本地 CPU

### 第四步：构造传输图

FlexKV 根据命中位置自动构图。

例如单机常见路径：

```text
SSD -> CPU -> GPU
CPU -------> GPU
```

如果开了 GDS，则 SSD 命中的部分可以变成：

```text
SSD -> GPU
```

少了一次 CPU 中转。

### 第五步：只计算没命中的尾巴

如果前面 100 个 block 命中，只剩后面 20 个 block 没命中，那么模型只需要继续算这 20 个 block。

这就是它真正省下来的算力。


## 6. 它的写入流程是怎样的

这里说的是“刚算出来的新 KV，怎么存起来给下次用”。

### 基本思路

新产生的 KV 一般先从 GPU 落到 CPU。

然后：

- CPU 层保留一份
- 如果开了 SSD，再写一份到 SSD
- 如果开了远端共享，再写到远端存储或发布元数据

### 一个关键细节：先插索引，但先不标 ready

FlexKV 在写入时会先把新节点插到索引里，但先标成 `is_ready = false`。

原因很直接：

- 索引可以先知道“这段前缀将要存在”
- 但在真实数据还没搬完之前，不能把它当成可读数据

等对应的数据搬运完成后，再通过回调把节点改成 ready。

这是一个很典型的“控制面先登记，数据面完成后再生效”的设计。


## 7. 它为什么比普通 KV Cache 方案更快

### 原因 1：把重复计算变成命中后搬运

Prefill 很贵。

如果一段前缀已经算过，那么重新算通常比“从 CPU/SSD 拉回 KV”更浪费。

FlexKV 本质上是在做一件事：

> 用 I/O 代替重复计算。

### 原因 2：命中是按前缀匹配，不是按整段请求匹配

如果只能做整段缓存，那么稍微改动一点点，整段就 miss 了。

FlexKV 用的是最长前缀匹配，所以哪怕请求只共享前半段，也能复用前半段的 KV。

### 原因 3：控制面和数据面分离

- 控制面负责“决定从哪儿拿”
- 数据面负责“把数据搬回来”

这样可以单独优化传输路径，而不是把所有逻辑都塞在推理框架主流程里。

### 原因 4：异步和并发

它不是同步搬运，而是用传输图和多个 worker 去执行。

可用的底层加速包括：

- GPU/CPU 间自定义 CUDA 传输核
- SSD 的 `io_uring`
- GDS 直通 SSD -> GPU
- Mooncake RDMA 跨节点传输

### 原因 5：分布式查询不依赖中心化在线查找

如果每次都查 Redis 才知道远端有没有 KV，那么延迟会非常难看。

FlexKV 的做法是：

- Redis 存元数据
- 每台机器本地维护一份远端索引快照

这样命中判断在本地内存里完成，快得多。


## 8. 跨节点复用是怎么做的

这是 FlexKV 比较有特点的一部分。

### 8.1 远端 CPU 命中

如果某些 KV 在另一台机器的 CPU 缓存里，那么本地可以直接通过 Mooncake 做 RDMA 读取。

可以理解成：

```text
远端 CPU 内存 -> 本地 CPU 内存
```

这是单边 RDMA read 路径。

### 8.2 远端 SSD 命中

如果 KV 在另一台机器的 SSD 上，流程会复杂一点：

1. 本地先发一个控制消息给远端
2. 远端把 SSD 上的数据读到自己的临时 CPU buffer
3. 远端再用 RDMA 把这段数据写回本地 CPU buffer
4. 最后远端发完成通知

这条路径本质上是：

```text
远端 SSD -> 远端 CPU 临时缓冲 -> 本地 CPU
```

它比“直接访问远端文件系统再自己读”更适合做高性能数据面。


## 9. 它真正依赖的关键数据结构

### 9.1 SequenceMeta

负责：

- 保存 token_ids
- 生成 block 级前缀 hash
- 记录 namespace 信息

### 9.2 RadixTree / CRadixTreeIndex

负责：

- 最长前缀匹配
- 插入新前缀
- 驱逐旧节点
- 维护节点状态

### 9.3 Mempool

负责：

- 分配可用物理 block
- 回收被驱逐或用完的 block

### 9.4 TransferOpGraph

负责：

- 把一次 get / put 拆成多个传输操作
- 表达先后依赖关系
- 交给 TransferEngine 调度执行


## 10. 这套方案适合什么场景

最适合这些情况：

- 有明显前缀复用的在线推理
- 长上下文、多轮对话
- RAG 模板很多、重复率高
- Agent 场景里系统提示和工具框架固定
- 多机服务里不同节点可能遇到相同前缀

如果请求几乎没有公共前缀，那么它的收益会小很多。


## 11. 它的局限也要说清楚

### 局限 1：前缀重用率低时，收益有限

如果每次请求都完全不同，FlexKV 也没有太多东西可复用。
 - KV cache 迁移 主要是在解决“GPU 放不下”的问题
  - 但 FlexKV 的收益 最终还是要靠“以后能不能再用上这份被迁移出去的 KV”

  所以更准确地说：

  > 迁移是手段，复用是回报。


### 局限 2：外部存储再快，也不可能比 GPU 显存快

所以它不是“外部缓存一定比 GPU 快”，而是“外部缓存通常比重新 Prefill 划算”。

### 局限 3：当前分布式匹配策略还不算最激进

从代码实现看，它目前更像是：

- 本地结果和远端结果分别算一遍
- 选匹配更长的一边

而不是把两边命中的片段拼成一个更长的全局最优前缀。

这说明这套系统还有继续优化空间。

### 局限 4：最后一个不完整 block 通常不会参与缓存

这是很多 block-based cache 方案都会有的现实约束。


## 12. 思路总结

可以这样说：

> FlexKV 像是在给大模型做“分层文件缓存 + 前缀索引”。
>  
> 模型算过的前缀不会直接丢掉，而是先登记索引，再存到 CPU、SSD 或别的机器上。
>  
> 下次再来相似请求时，先查索引看前面哪些内容已经算过；
> 如果找到了，就把已经算好的中间结果拿回来继续用，只计算没命中的剩余部分。


## 13. 方法总结

FlexKV 用了这些关键方法：

- 块级前缀哈希：把请求转成可比较的前缀签名
- RadixTree 最长前缀匹配：快速找到已经算过的 KV
- 多级缓存：把 KV 放在 CPU / SSD / Remote，而不是只放 GPU
- 异步传输图：把读取和写入拆成可并发执行的传输任务
- 驱逐策略：决定缓存满了以后淘汰谁
- 分布式索引快照 + Lease：让多节点之间能安全复用 KV
- io_uring / GDS / RDMA：把“拿回 KV”这件事做得尽量快

它解决的核心问题是：

> 少重复算前缀，少浪费 GPU 显存，少让多机重复工作。
