# 基线测试报告

## 1. 背景

本项目围绕赛题中的 KV Cache 分层管理问题展开。赛题核心关注点不是模型能力分数，而是：

- 长上下文下 KV Cache 带来的显存压力
- 缓存命中与复用是否真实发生
- 吞吐、延迟与资源占用是否可接受
- 后续 KVCache/offload 优化是否能在原始系统之上形成可量化收益

因此，在进入 KVCache/offload 版本开发前，需要先建立原始 vLLM 的基线结果。该基线用于回答两个问题：

1. 原始系统在不同场景下的性能与资源表现如何
2. 后续优化版相对原始系统到底提升了什么

本轮工作目标是完成原始 vLLM 的 baseline 测试，并形成一份可复现、可对比、可汇报的结果基线。

## 2. 环境

### 2.1 推理环境

- 推理框架：`vLLM 0.19.1`
- 模型：`Qwen3-8B`
- 模型路径：`/root/autodl-tmp/Qwen/Qwen3-8B`
- 服务名：`qwen3-8b`
- 推理方式：云服务器本机部署、本机调用

说明：

- 当前测试不是调用外部第三方 API
- 模型文件位于云服务器本地磁盘
- 推理在云服务器本地 GPU 上执行

- `python vllm_model.py` 路径可正常加载模型并生成结果

### 2.3 采集方式

本轮基线采用两类测试方式：

1. 官方 benchmark

- `vllm bench serve`
- 用于建立固定输入长度的纯性能参考线

2. 自定义场景脚本

- `tests/vllm_baseline_runner.py`
- 用于跑五组场景 baseline
- 自动采集 `/metrics`、GPU 指标、CPU 内存和系统负载

- 输入长度增长时原始系统纯性能如何退化
- 后续优化版与原始系统相比时，性能趋势是否改善

当前已完成：

| 模型 | 输入长度 | 输出长度 | 请求数 | RPS | 成功率 | req/s | output tok/s | Mean TTFT | P99 TTFT | Mean TPOT |
|---|---|---|---|---|---|---|---|---|---|---|
| qwen3-8b | 2048 | 128 | 20 | 2.0 | 100% | 1.73 | 220.95 | 291.88 ms | 621.65 ms | 15.66 ms |
| qwen3-8b | 8192 | 128 | 20 | 1.0 | 100% | 0.88 | 112.53 | 1178.67 ms | 2934.98 ms | 33.56 ms |

说明：

- 这组结果属于固定长度性能参考线
- 它不是五组场景 baseline 的主体，但适合做长度退化趋势说明

## 3.2 数据集与五组场景

为了覆盖赛题所需的缓存、长上下文和混合负载行为，本轮 baseline 使用五组场景：

| 分组 | 主要目标 | 数据来源 |
|---|---|---|
| 短上下文基础组 | 验证基础功能与监控链路 | 合成短 prompt |
| 中长上下文主测试组 | 验证常规真实长文本场景 | LongBench `qasper`（本地 JSONL） |
| 超长上下文极限组 | 验证超长输入压力 | `ultra_long_synth` |
| 共享前缀命中组 | 验证 prefix cache 查询与命中 | 自合成共享前缀数据 |
| 混合压力组 | 验证不同长度请求混跑稳定性 | 合成混合负载 |

另外补充了一组开源超长长文本结果：

| 补充组 | 作用 | 数据来源 |
|---|---|---|
| 开源超长补充结果 | 补充真实开源超长文本输入 | `NeedleBench / en_haystack_texts` |

说明：

- `NeedleBench / multi_needle_reasoning_needle` 曾被尝试，但样本过短，仅作为接入验证保留，不作为最终主结果
- `LongBench` 使用本地 JSONL 方式读取，以避免在线依赖带来的不稳定性
- `ultra_long_synth` 虽然是 synthetic，但能稳定制造超长输入压力，因此被保留为当前阶段超长组主结果

## 5. 数据分析

## 5.1 长上下文与混合负载会提高缓存占用

从 `kv_cache_usage_max` 看，当前各组压力呈现如下趋势：

- 短上下文基础组：`0.010176`
- 共享前缀命中组：`0.025259`
- 超长上下文极限组（主结果）：`0.047429`
- 中长上下文主测试组：`0.058514`
- 开源超长补充结果：`0.062693`
- 混合压力组：`0.083772`

这说明：

- 在当前已测场景中，场景越复杂，KV cache 占用越高
- 混合压力组的缓存占用相对最高，说明不同长度请求混跑更容易放大缓存与调度压力
- 但从绝对值看，当前最高值约为 `8.38%`，仍未接近缓存池上限

结合赛题背景，这一点很重要，因为赛题核心问题本来就是：

- 长上下文场景下，KV Cache 会持续增长
- 需要通过更好的缓存管理和迁移机制缓解压力

当前基线已经证明原始系统确实存在这种压力。

## 5.2 共享前缀场景已验证命中真实发生

共享前缀组最终结果为：

- `prefix_queries_delta = 21002.0`
- `prefix_hits_delta = 20960.0`

混合压力组中也出现：

- `prefix_queries_delta = 34311.0`
- `prefix_hits_delta = 34208.0`

这说明：

- 共享前缀场景本身构造有效
- 当前服务下 prefix cache 查询与命中已经真实发生
- 后续如果做 KVCache/offload，对 prefix reuse 场景的收益是有机会被观测到的

这与赛题中“监控访问模式、提高命中率”的目标是直接对齐的。

## 5.3 开源数据与 synthetic 数据各有作用

当前结果里，`LongBench` 和 `NeedleBench / en_haystack_texts` 提供了开源数据支撑，而 `ultra_long_synth` 提供了更稳定的超长输入压力。

具体来说：

- `LongBench`
  - 提供真实中长文本场景
  - 适合作为常规长上下文 baseline
- `NeedleBench / en_haystack_texts`
  - 提供开源超长长文本补充
  - 适合作为“真实开源超长结果”说明
- `ultra_long_synth`
  - 长度可控
  - 更适合作为当前阶段超长上下文极限组主结果

这和当前环境限制也有关：

- 需要保证实验稳定可跑
- 同时又不能完全失去开源数据支撑

因此当前“主结果 + 开源补充结果”的组合是合理的。

## 5.4 固定长度 microbenchmark 仍然有价值

虽然固定长度参考线不是五组场景 baseline 的主体，但它补充回答了：

- 输入从 `2K` 提升到 `16K` 时，原始系统的吞吐和延迟如何退化

结果显示：

- req/s 从 `1.73` 下降到 `0.41`
- Mean TTFT 从 `291.88 ms` 上升到 `3105.55 ms`
- Mean TPOT 从 `15.66 ms` 上升到 `48.97 ms`

这说明：

- 单纯输入长度增长，就足以让原始系统明显变慢
- 这条参考线能帮助解释后续优化到底缓解了多少“长度增长带来的天然退化”

## 6. 测试结论

结合赛题背景和当前环境，本轮 baseline 可以得出以下结论：

1. 原始 vLLM 服务已经能够稳定支撑本轮五组 baseline 测试

- 所有最终采用结果成功率均为 `100%`
- 当前服务和外部采集链路稳定可用

2. 长上下文和混合负载会增加缓存压力

- `kv_cache_usage_max` 在混合压力组达到 `0.083772`，即约 `8.38%`
- 说明缓存压力在复杂场景下会更加明显
- 但从绝对值看，当前测试尚未把缓存池逼近上限

3. 共享前缀场景下的命中已被实际观测到

- prefix cache 查询和命中都已经有明确增量
- 后续缓存优化收益有明确观测入口

4. 当前阶段已经具备后续对比测试的基线条件

因为现在已经同时具备：

- 固定长度性能参考线
- 五组比赛场景 baseline
- 开源长文本与超长文本补充结果
- 统一的过程性采集指标

因此后续进入 KVCache/offload 版本时，可以直接在同一套场景和指标口径下进行对比。
