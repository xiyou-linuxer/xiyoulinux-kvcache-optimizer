## 1. 先说结论

LMCache 不是一个“把 Attention 算子本身改快”的项目。

它做的事情更像是：

- 把已经算出来的 KV Cache 当成一批可以缓存、查找、预取、回填的==大对象==
- 尽量避免同样内容被重复 Prefill
- 当 GPU 放不下时，把 KV 放到 CPU、磁盘和远端后端里
- 当下一次又需要这些内容时，先把旧 KV 取回来，再只补算缺失部分
- 在普通前缀复用之外，还尝试支持段落级、非前缀位置的复用

一句话概括：

> LMCache 的核心思想是“把 KV Cache 变成一批可寻址对象，先查已有对象，再异步预取和回填到 GPU，从而减少重复计算和等待时间”。

---

## 2. 它到底解决了什么问题

LMCache 主要想解决三类问题。

### 问题 1：GPU 里的 KV 留不住

即使推理引擎已经支持前缀缓存，也还是会遇到：

- GPU 显存有限
- 已经算过的 KV 很快被驱逐
- 下次命中时仍然可能重算

LMCache 想做的是：

- 不让旧 KV 只留在 GPU
- 而是继续保存到 CPU、磁盘或远端存储

### 问题 2：只做 GPU 内部前缀复用还不够

很多系统原本只能做到：

- GPU 里还有，就复用
- GPU 里没了，就重新算

LMCache 更进一步：

- GPU 没有，不代表真的没有
- 还可以看 CPU、Disk、Remote 有没有

### 问题 3：真实场景下不一定只是严格前缀重复

在 RAG 场景里，经常会出现：

- 一批资料块以前都见过
- 但这次拼接顺序变了
- 不是严格的“前半段完全一致”

普通 prefix cache 对这类情况支持有限。LMCache 通过 blending 机制，尝试让这类场景也能复用一部分旧 KV。

所以 LMCache 真正解决的是：

> 不只是“有没有前缀缓存”，而是“怎样把旧 KV 存成可查找对象，并在 prefix 和非 prefix 的复用场景里都尽量重新利用它们”。

## 3. 它用了哪些核心算法和机制

LMCache 不是靠一个单独算法提速，而是靠几种机制配合。

### 3.1 Chunk 化对象管理

这是 LMCache 最基础的一步。

它默认不会把整段请求当成一大块，也不会按单个 token 管理，而是按固定大小切成 chunk。

默认配置里：

- `chunk_size = 256`

也就是说，它更喜欢把 KV Cache 管成一批“256 token 一组”的对象。

这样做的好处是：

- 对象粒度稳定
- 适合做缓存键
- 适合写到 CPU / Disk / Remote
- 适合批量预取和批量回填

### 3.2 前缀相关哈希

在默认的 `ChunkedTokenDatabase` 里，LMCache 也是做前缀相关哈希的。

意思是：

- 第一个 chunk 的 hash 依赖前 1 个 chunk
- 第二个 chunk 的 hash 依赖前 2 个 chunk
- 依次类推

所以在默认模式下，它仍然具备“最长前缀命中”的语义。

这说明 LMCache 并不是完全抛弃 prefix cache，而是在 prefix cache 基础上继续扩展。

### 3.3 Segment 切分

[[###3.7 Cache Key：为什么说它一开始就可寻址|3.7 Cache Key：为什么说它一开始就可寻址]]

- chunk = 按长度切
- segment = 按语义/分隔符切


这是 LMCache 比很多普通 KV cache 方案更特别的地方。
blending 你可以先简单理解成：

  > 把多段已经缓存过的 KV 尽量拼起来一起用，而不是只会复用严格前缀。

  它是 LMCache 里专门为了处理 RAG / 多段资料拼接 这种场景加的机制。


如果开启 blending，它不再只按固定 chunk 切，而是按分隔符把输入切成多个 segment。

  LMCache 想把“整段前缀复用”变成“按段落/块复用”。

  你可以先对比普通前缀缓存。

  #### 1. 普通前缀缓存怎么复用

  普通前缀缓存要求：

  - 从开头开始一段内容完全一样
  - 才能复用这段 KV

  比如：

  请求A: system + chunk1 + chunk2 + chunk3 + questionA
  请求B: system + chunk1 + chunk2 + chunk3 + questionB

  这里前面 system + chunk1 + chunk2 + chunk3 一样，所以能复用。

  但如果变成：

  请求A: system + chunk1 + chunk2 + chunk3 + questionA
  请求B: system + chunk2 + chunk1 + chunk3 + questionB

  普通前缀缓存就很难用了，因为从 chunk1 那里开始顺序就不同了。

  这里的 chunk1 / chunk2 / chunk3 其实不是在说 LMCache 的 fixed-size chunk。
  这里更接近是在说：

  - 文档块 1
  - 文档块 2
  - 文档块 3

  也就是 RAG 里的资料段落。

  为了避免混淆，这里其实更应该写成：

  system + segment1 + segment2 + segment3 + question


  ———

  ####  2. LMCache blending 想做什么

  它的思路是：

  - 不把整段 prompt 只看成“一整条前缀”
  - 而是先按特殊分隔符切成多个 segment

  比如切成：

  - system prompt
  - chunk1
  - chunk2
  - chunk3
  - question

  然后：

  - 每个 segment 单独生成自己的 key
  - 单独缓存对应的 KV

  这样下一次请求来时，即使顺序变了，只要某些 segment 本身还是同一个，就还能单独
  命中。

  你可以把它理解成：

  > 普通前缀缓存像“整段句子一起复用”，
  > blending 更像“把句子拆成积木，每块积木单独复用”。

  ———

  ####  3. 为什么“顺序变了”还能复用

  因为 LMCache 记住的不是：

  - “这一整条 prompt 的前缀”

  而是：

  - “chunk1 这块内容以前见过”
  - “chunk2 这块内容以前见过”
  - “chunk3 这块内容以前见过”

  所以即使第二次顺序换成：

  system + chunk2 + chunk1 + chunk3 + question

  它仍然可以说：

  - system 命中
  - chunk2 命中
  - chunk1 命中
  - chunk3 命中

  也就是按段去找旧 KV，而不是只按“整段前缀”去找。



### 3.4 Lookup + Prefetch

LMCache 不想等到“模型马上要用 KV 的那一刻”才开始找和搬数据。
  它想提前做准备。

  你可以把 lookup + prefetch 拆成两步看。

  ———

  ####  1. lookup 是什么

  lookup 就是先问一句：

  > “我要的这些旧 KV，缓存里到底有没有？”

  比如一个请求来了，LMCache 会先把它切成 chunk/segment，然后检查：

  - chunk1 有没有
  - chunk2 有没有
  - chunk3 有没有

  最后得到一个结果，比如：

  - 前 3 个 chunk 都有
  - 后 2 个没有

  这一步本质上只是“查”。

  ———

  #### 2. retrieve 是什么

  retrieve 才是真正把旧 KV 拿回来。

  也就是说：

  - lookup = 查库存
  - retrieve = 真正搬货

  如果只做 lookup，不做 retrieve，那系统只是“知道有货”，但还没把货搬到 GPU。

  ———

  #### 3. 那 prefetch 又是什么

  prefetch 的意思是：

  > 在真正需要这些 KV 之前，提前开始把它们往近处搬。

  比如旧 KV 在磁盘或远端机器上。
  如果你等到模型已经卡在那里、马上要用时才开始搬，那还是会等很久。

  所以更聪明的做法是：

  - 一边查
  - 一边就开始准备把数据拉到本地 CPU 或本地缓存层

  这样等真正执行 retrieve 时，很多数据已经更靠近 GPU 了。

  ———

  #### 4. 为什么说 LMCache 把 lookup 和 prefetch 连在一起

  因为它不是这样做的：

  第 1 步：先查一下有没有
  第 2 步：过一会儿再决定要不要开始搬

  而是更像这样：

  第 1 步：一查到有命中
  第 2 步：就顺手开始预取

  也就是说：

  - 查有没有
  - 和
  - 开始准备搬回来

  这两件事尽量一起触发。


  ———

  #### 5. 用最简单例子讲

  假设：

  - 旧 KV 在远端磁盘上
  - 新请求来了
  - LMCache 先做 lookup

  ### 普通做法

  1. lookup 告诉你：有
  2. 系统先继续别的流程
  3. 等模型真正需要这份 KV 时，才开始从远端磁盘搬
  4. 模型开始等

  ### LMCache 的思路

  1. lookup 告诉你：有
  2. 立刻触发 prefetch
  3. 把这份 KV 提前往本地 CPU / 本地缓存层搬
  4. 等真正 retrieve 时，数据已经近很多了

  所以区别就在于：

  - 普通做法：先知道，再等到最后才搬
  - LMCache：一知道就提前搬


lookup 做的事情包括：

- 查已经有哪些 chunk/segment 在缓存里
- 算出命中了多少 token
- 在异步路径里顺便启动 prefetch

也就是说，它把“查有没有”和“开始准备取回来”这两步尽量连在一起。

### 3.5 Layerwise 管线

这也是 LMCache 非常重要的优化点。

普通做法是：

- 整份 KV 全部取回来
- 然后模型才开始继续 forward

Layerwise 的思路是：

- 第 0 层先回来，模型先跑第 0 层
- 第 1 层在后台继续加载
- 上一层存储和下一层计算也能重叠

这意味着 LMCache 不只是减少“重算量”，还在想办法隐藏“回填延迟”。

### 3.6 多级后端

LMCache 的后端体系很丰富，典型包括：

- LocalCPUBackend
- LocalDiskBackend
- RemoteBackend
- P2PBackend
- PDBackend
- GdsBackend

这说明它的思路不是“把 KV 只存到某一个地方”，而是让不同层、不同协议、不同部署方式都能挂进来。

### 3.7 Chunk-key查询

默认 chunk 模式不是每个 chunk 完全独立生成 key，而是按前缀链式生成：第 n
  > 个 key 表示“前 n 个 chunk 的前缀状态”；只有 blending 的 segment 模式，才更接
  > 近每个段单独生成 key。

这是 LMCache 和很多“只把 KV 当成 GPU 里一片内存”的方案不太一样的地方。



> 在 LMCache 里，一段 KV 一旦被切成 chunk 或 segment，就会很快被转换成一个 `CacheEngineKey`，后面系统就是围绕这个 key 去查、取、存、搬。

这里的意思不是说“只要一段 token 就有一个纯内容地址”，而是说：

- 这段 chunk/segment 很快会有一个统一对象标识
- 以后无论它在 CPU、Disk 还是 Remote，都可以按这个 key 去找

这个 key 不是只包含内容哈希，它通常还带有：

- 模型信息
- world_size
- worker_id
- dtype
- 请求标签信息

所以更准确地说，它是“对象标识”，不是单纯的“内容地址”。

另外还要注意：

- 默认 `chunk` 模式下，它仍然保留前缀语义，因为 hash 是前缀相关的
- 只有在 blending 的 `segment` 模式下，它才更明显地表现为“按语义段独立对象寻址”

所以 LMCache 的一个重要特点可以概括成：

> 它不只是管理“GPU 上某块内存”，而是把 KV 管理成一批带 key 的对象。

### 3.8 冷热缓存是怎么理解的

LMCache 里也有冷热缓存的概念，但它不是那种复杂预测模型。

更准确地说，它的冷热主要由这几类因素共同决定：

#### 第一层：对象现在在哪一层

默认可以这么理解：

- `LocalCPUBackend`：更热
- `LocalDiskBackend`：更冷
- `RemoteBackend / P2PBackend`：更偏远、更冷

也就是说，冷热首先体现在“对象现在离 GPU 有多近”。

#### 第二层：最近是否被访问过

LMCache 有 `cache_policy`，默认通常是 `LRU`。

这意味着：

- 最近刚命中过的对象，更像热点
- 很久没命中的对象，更像冷数据

#### 第三层：对象当前有没有被 pin / lock

如果一个对象正在 lookup、prefetch、retrieve 或 store 过程中，它不只是“值不值得留”，还涉及“当前能不能动”。

所以：

- 被 pin 的对象更像临时热点
- 被 lock 保护的对象不能随便驱逐

#### 第四层：容量压力

当本地热层空间不够时，LMCache 才会真正把这些冷热判断变成驱逐动作。

所以最准确的说法是：

> LMCache 的冷热管理主要不是靠复杂预测，而是靠“层级位置 + 命中更新 + 容量限制 + pin/lock 保护”来共同决定。

### 3.9 什么叫活跃后端

“活跃后端”这个词，意思其实很简单：

> 当前这个 LMCache 实例里，真正启用、可用、并会参与查找或存储的那些后端。

比如你可能配置了：

- LocalCPUBackend
- LocalDiskBackend
- RemoteBackend

那这些就是当前活跃后端。

但如果：

- 某个后端没启用
- 某个后端被 bypass 掉了
- freeze 模式下它不允许参与查找
- 本次 `search_range` 没把它纳入

那它就不会参与这次 lookup / retrieve / store。

所以“按活跃后端顺序查”真正的意思是：

- 不是查所有理论上的后端
- 而是只查当前真正可用、没被排除的那几层

这比写死成“固定 L1 → L4”更准确。

### 3.7 CacheBlend

> segment 是切分方式
  > CacheBlend 是利用这些 segment 去做复用的机制
  
  

这是 LMCache 最有辨识度的算法特性之一。

它的目标是：

- 不是只复用前缀
- 而是复用输入中的多个已知段落
- 即使它们这次不是按原来的顺序出现

当然，它并不是完全零成本做到这件事，而是会：

- 对一部分 token 做重新计算
- 用 `blend_recompute_ratio` 等参数控制精度和代价

所以你可以把 CacheBlend 理解成：

> “段落级复用 + 局部重算修正”

### 3.8 压缩序列化

LMCache 还支持把 KV 用 `cachegen` 这种方式压缩后再写到远端。

这主要解决的是：

- 远端传输成本
- 远端存储体积

所以它不仅想“把 KV 存出去”，还想让“存出去”这件事更便宜。

## 4. 一个简单例子：LMCache 为什么不只做前缀复用

假设有一个 RAG 场景。

第一次请求：

```text
system + chunk1 + chunk2 + chunk3 + questionA
```

第二次请求：

```text
system + chunk2 + chunk1 + chunk3 + questionB
```

如果只做严格前缀匹配，会发生什么？

- system 一样，能复用
- 后面一旦顺序不一样，很多普通 prefix cache 就很难继续利用

但 LMCache 会尝试这样做：

- 把 `system`、`chunk1`、`chunk2`、`chunk3` 都看成独立对象
- 这次虽然顺序变了
- 但对象本身以前已经出现过
- 所以还能把这些旧 KV 重新拼起来用

当然，因为上下文顺序变化，不能完全无脑复用。

所以它会：

- 复用大部分旧段落 KV
- 再对一小部分 token 重新计算，修正误差

这就是 LMCache 为什么说自己不只做 prefix reuse 的原因。

## 5. 它的查找和取回流程是怎样的

这里说的是：一份旧 KV 已经存在了，LMCache 怎么把它重新拿回来。

### 第一步：先把输入变成对象键

默认模式下：

- 把输入按 `chunk_size` 切成 chunk
- 每个 chunk 生成一个前缀相关 hash
- 再封装成 `CacheEngineKey`

blending 模式下：

- 按分隔符切成 segment
- 每段生成自己的 key

### 第二步：lookup

LMCache 先做 lookup，而不是直接 retrieve。

lookup 的作用是：

- 查哪些 chunk/segment 已存在
- 算出命中了多少 token
- 可选地 pin 住这些对象
- 异步模式下顺便触发 prefetch

### 第三步：prefetch

如果开启 async lookup client，worker 侧会提前把命中的对象准备到本地。

你可以理解成：

- scheduler 还在等结果
- worker 已经在后台把对象往近处拉了

这样真正 retrieve 时会更快。

### 第四步：retrieve

真正 retrieve 时，LMCache 会：

- 从 StorageManager 找到这些 key 对应的 `MemoryObj`
- 生成一个 `ret_mask`
- 再调用 GPU connector 的 `batched_to_gpu()` 把对象写回 GPU

### 第五步：只补算缺失部分

一旦旧 KV 被注入回 GPU，模型只需要计算没命中的那部分。

所以 retrieve 的核心目的不是“简单读文件”，而是：

> 把已经存在的 KV 对象重新回填到模型执行现场。

---

## 6. 它的写入流程是怎样的

这里说的是：新算出来的 KV 怎么被 LMCache 保存下来。

### 第一步：切对象

LMCache 先根据 token 和 mask，把本次可保存的部分切成 chunk 或 segment。

### 第二步：分配本地对象

StorageManager 会为这些 chunk 分配 `MemoryObj`。

你可以把它理解成：

- 先准备一批装 KV 的容器

### 第三步：从 GPU 拉下来

然后 LMCache 会调用 GPU connector 的：

- `batched_from_gpu()`

把新算出的 KV 从 GPU 复制到这些 `MemoryObj` 里。

### 第四步：写入各层后端

接下来通过 `batched_put()`，把对象异步写到配置好的后端：

- CPU
n- Disk
- Remote
- 或其他 backend

### 第五步：响应先返回，写入走后台

这是 LMCache 很重要的一个设计点。

它不会要求：

- “所有缓存都写完了，用户才能拿到输出”

而是更偏向：

- 输出先返回
- 缓存后台慢慢下沉

这样能减少对在线响应时间的影响。

## 7. 它为什么比普通 KV Cache 方案更快

### 原因 1：把重复计算变成旧对象回填

普通情况下，如果 GPU 里没有旧 KV，就重新算。

LMCache 的思路是：

- 先查这些旧对象是不是已经在 CPU/Disk/Remote
- 有就取回来
- 只算缺失部分

所以它本质上还是在做：

> 用 I/O 代替重复计算。

### 原因 2：它把查找和预取拆开了

很多方案只在真正 retrieve 时才开始处理。

LMCache 更进一步：

- 先 lookup
- 提前 prefetch

这意味着当引擎真正需要这些 KV 时，很多准备工作已经做了一部分。

### 原因 3：Layerwise 能隐藏一部分加载时间

这是 LMCache 很大的特色。

它不是等整份 KV 全部就绪后才继续 forward，而是尽量做到：

- 第 0 层回来就先算第 0 层
- 第 1 层在后台继续拉

也就是说，它不只是减少了工作量，还试图隐藏加载延迟。

### 原因 4：支持非前缀复用

在一些 RAG 场景下，真正的收益点不只是共享前缀，而是共享资料段落。

LMCache 的 blending 让它有机会复用：

- 这次顺序变了，但内容以前已经见过的那些块

这使它在复杂拼接场景下比单纯 prefix cache 更有想象空间。

### 原因 5：后端体系更灵活

它不是绑定某一个本地存储层，而是能根据部署方式挂：

- CPU
- Disk
- Remote
- P2P
- PD
- GDS

所以它更容易在不同环境里调整为合适的数据路径。

---

## 8. 非前缀复用是怎么做的

这是 LMCache 最需要单独讲的一部分。

### 普通前缀缓存有什么限制

它通常要求：

- 请求前面一大段完全一样
- 才能直接复用这段 KV

一旦内容顺序变化，普通 prefix cache 往往就帮不上太多忙。

### LMCache 的思路是什么

LMCache 通过 `SegmentTokenDatabase + CacheBlend` 试图处理这种场景。

大致逻辑是：

1. 用特殊分隔符把 prompt 切成多个段
2. 每段单独缓存
3. 下一次 prompt 虽然顺序不同，但如果段本身相同，仍然可以命中这些旧段
4. 因为重新拼接后上下文关系变了，LMCache 会再对一部分 token 做重算修正

### 为什么不能完全免费

因为 Transformer 的 KV 不是孤立段落向量，仍然受上下文影响。

所以 LMCache 不能像拼积木一样把所有旧 KV 直接无损拼起来。

它的做法是：

- 能复用的先复用
- 可能不准确的部分再补算

所以这个机制的本质是：

> 不是“完全摆脱前缀约束”，而是“在非前缀场景里尽可能多复用，再用局部重算修正”。

---

## 9. 分布式共享是怎么做的

LMCache 的分布式路径更偏“服务中间层”而不是单纯本地缓存。

### 第一层：lookup client / lookup server

在 vLLM 集成里，scheduler 和 worker 之间会通过 lookup client / lookup server 协作。

也就是说：

- scheduler 发 lookup 请求
- worker 负责真正查后端
- 异步模式下还会触发 prefetch

### 第二层：StorageManager + Backend

真正的数据可能在：

- LocalCPUBackend
- LocalDiskBackend
- RemoteBackend
- P2PBackend
- PDBackend

StorageManager 统一管理这些后端。

### 第三层：L1 / L2 控制器

LMCache 设计文档里明确把分布式缓存管理拆成：

- `StoreController`
- `PrefetchController`
- `L1`
- `L2Adapter`

这说明它不是简单“远端拿一把”，而是有显式控制器在管理：

- 什么写入 L2
- 什么从 L2 预取到 L1
- 哪些对象加锁
- 哪些对象释放

### 第四层：P2P / PD / NIXL

如果部署需要更高性能传输，它还可以结合：

- P2PBackend
- PDBackend
- transfer channel（如 NIXL）

所以 LMCache 的分布式实现，不只是“远端对象存在”，而是：

> lookup、prefetch、锁管理、后端对象读取和高性能传输共同组成的一套服务流程。

---

## 10. 它真正依赖的关键数据结构

### 10.1 TokenDatabase

负责：

- 把输入 token 变成 chunk 或 segment
- 生成 hash / key

这是 LMCache 的对象切分入口。

### 10.2 CacheEngineKey / LayerCacheEngineKey

负责：

- 给每个对象提供稳定唯一标识
- 支持 layerwise 按层寻址
- 支持请求标签隔离

你可以把它理解成：

- `CacheEngineKey`：普通 chunk/segment 对象的统一地址
- `LayerCacheEngineKey`：在 layerwise 模式下，每一层对象的单独地址

所以 LMCache 很多操作本质上都在围绕 key 进行：

- lookup key
- prefetch key
- retrieve key
- store key

### 10.3 MemoryObj

负责：

- 作为 KV 在 CPU / Disk / Remote 层流动时的统一对象封装

可以把它理解成“KV 的标准运输容器”。

### 10.4 LMCacheEngine

负责：

- lookup
- retrieve
- store
- lookup_unpin
- async_lookup_and_prefetch

它是主控制面。

### 10.5 StorageManager

负责：

- 统一调度各层 backend
- allocate / get / put / prefetch
- 管理 allocator backend

### 10.6 GPUConnector

负责：

- `batched_from_gpu()`
- `batched_to_gpu()`

也就是 GPU 和外部层之间真正的数据搬运。

## 11. 这套方案适合什么场景

最适合这些情况：

- 长上下文场景
- 多轮对话
- RAG
- 需要把 KV 从 GPU 扩展到 CPU/Disk/Remote
- 分离式 prefill/decode 场景
- 希望通过异步 lookup/prefetch 降低 TTFT 的场景

如果你的业务只在单机 GPU 内部做简单 prefix cache，而且没有更复杂的远端层和对象共享需求，那么 LMCache 的很多能力可能发挥不出来。

---

## 12. 它的局限也要说清楚

### 局限 1：系统复杂度明显更高

LMCache 不只是一个简单缓存点子，而是一套：

- engine adapter
- token database
- key 体系
- storage manager
- 多 backend
- lookup/prefetch server

组合起来的系统。

所以它的部署和调试成本比单纯 GPU 内 prefix cache 大很多。

### 局限 2：收益依赖于内容重复和后端性能

如果请求之间几乎没有可复用内容，那么 lookup、prefetch、store 这些额外动作的收益会下降。

### 局限 3：非前缀复用不是完全免费

CacheBlend 不是“零成本把乱序段落全部无损拼起来”，而是：

- 尽量复用
- 再局部重算修正

所以它本身仍然有代价，也需要参数调优。

### 局限 4：远端层再快，也不可能比 GPU 常驻更快

LMCache 的意义不是“远端比 GPU 更快”，而是：

- 当 GPU 留不住时
- 远端回填通常比完全重算更划算

---

## 13. 思路总结

如果把 LMCache 的主线压成几句话，可以这样理解：

1. 它先把 KV Cache 管成一批对象，而不是只管“当前 GPU 里有没有”。
2. 它先 lookup，再 prefetch，再 retrieve，而不是等真正需要时才临时查。
3. 它不只支持 prefix reuse，还尝试支持 segment 级非前缀复用。
4. 它最有特色的优化之一是 layerwise pipeline，也就是按层隐藏一部分回填延迟。
5. 它的目标是让旧 KV 离开 GPU 后仍然能继续发挥价值。

---

## 14. 方法总结

如果只看方法，可以压缩成下面这张表。

- Chunk 化：把 KV 管成稳定对象，方便缓存和传输。
- 前缀相关哈希：支持默认模式下的前缀连续命中。
- Segment 切分：支持 blending 场景下的段落级复用。
- Lookup + Prefetch：提前发现旧对象并提前准备。
- Retrieve + GPU 注入：把旧对象重新送回模型的 KV cache。
- Store + 异步写出：把新 KV 后台下沉到多级后端。
- Layerwise pipeline：让加载/存储和 forward 尽量重叠。
- CacheBlend：让非严格前缀场景也能复用一部分旧 KV。
- 多级 backend：让 CPU、Disk、Remote、P2P、PD、GDS 都能参与进来。
