# 比赛导向测试指标设计说明
现在的测试框架可以直接概括成一句话：

原始 vLLM 服务 + 官方 benchmark + 外部监控采集 + 分组数据集
## 1. 文档目的

本文档用于重新定义当前项目的测试指标体系。  
这里不机械照搬任务截图中的所有指标，而是结合以下三件事来收敛：

- 比赛题目的核心目标
- 你们当前项目的实际开发阶段
- 当前能够低成本、稳定采集到的数据

本文档主要回答两个问题：

1. 现在是不是一定要先用 vLLM 自带监控
2. 这个阶段到底应该测哪些指标

## 2. 先说结论

### 2.1 不是“只能用 vLLM 自带监控”

不是必须“只用 vLLM 自带监控”，但当前阶段**应该优先复用 vLLM 已有指标**，原因是：

- 成本最低
- 风险最小
- 先天适合原始 vLLM baseline
- 便于后续和优化版统一对比

所以更准确地说法是：

- `vLLM 自带监控` 是第一层
- `外部资源采集` 是第二层
- `你们方案专属埋点` 是第三层

当前阶段先做前两层，后续再补第三层。



## 3. 比赛题目真正关心什么

从赛题要求来看，核心目标不是模型答题分数，而是 KV Cache 系统行为。

可以收敛成四个核心问题：

### 3.1 是否缓解了长上下文下的显存压力

这是比赛主矛盾。

要回答的问题包括：

- 长上下文下显存是否快速增长
- 原始系统在哪个长度开始明显吃紧
- 后续 offload 是否能支持更长上下文

### 3.2 是否减少了因 KV 访问和迁移带来的等待

比赛明确提到：

- 最大化吞吐率
- 最小化 GPU 因等待 KV 迁移产生的等待时间

所以不能只看“能不能跑”，还要看：

- 吞吐有没有掉很多
- 延迟有没有被拖垮
- 回填/迁移是不是成了新瓶颈

### 3.3 是否真的发生了缓存命中和合理迁移

比赛里有：

- 监控访问模式
- 定义热度
- 迁移热数据/温数据/冷数据

所以不能只看系统层性能，还要看：

- 命中有没有发生
- block 是否被正确卸载和回填
- 迁移是否合理

### 3.4 是否能在少侵入条件下接入主流推理框架

这意味着你们的测试方案也要考虑：

- 尽量复用框架已有指标
- 新增监控尽量少侵入
- 不把测试方案建立在大量魔改源码上

## 4. 按阶段设计指标，而不是一次全上

当前阶段最合理的做法，是把指标分成三层。

## 4.1 第一层：当前必须测的 baseline 指标

这一层用于回答：

- 原始 vLLM 在长上下文下性能如何退化
- 当前基础监控链路是否可用
- 后续优化有没有对照基线

这一层是今天就应该测的。

## 4.2 第二层：当前应该补的扩展指标

这一层用于回答：

- 资源压力到底有多大
- prefix 命中是否发生
- 长上下文和共享前缀场景能否被观测

这一层应该尽快补，但不要求深入修改源码。

## 4.3 第三层：后续 KVCache 版本专属指标

这一层用于回答：

- offload/reload 到底发生了多少次
- GPU hit / CPU hit / miss 如何分布
- 迁移延迟和收益是否合理

这一层要在你们自己的 KVCache/offload 代码接入后再做。

## 5. 当前阶段推荐指标体系

下面是按“必须测 / 应补充 / 后续再补”整理后的推荐指标体系。

## 5.1 当前必须测

这些指标是当前阶段最重要的，因为它们直接决定 baseline 是否成立。

### A. 服务可用性指标

- 请求总数
- 成功请求数
- 失败请求数
- 成功率

为什么必须测：

- 连成功率都不稳定，后面的吞吐和延迟都不可信

### B. 核心性能指标

- Request throughput
- Output token throughput
- Mean TTFT
- P99 TTFT
- Mean TPOT
- P99 TPOT

为什么必须测：

- 这组指标最直接回答“长上下文是否拖慢系统”
- 能直接支撑 baseline 对比

### C. 输入长度分档

- 2K
- 8K
- 16K

为什么必须测：

- 比赛主线是长上下文和显存压力
- 不做长度分档，结果解释力很弱

### D. 模型与环境记录

- 模型名称
- tokenizer
- 最大上下文长度
- vLLM 版本
- GPU 型号
- 启动参数

为什么必须测：

- 模型和环境直接影响 KV 压力
- 不记录这些，结果不可复现

## 5.2 当前应该补充

这些指标很重要，而且当前阶段已经可以较低成本补上。

### A. 资源指标

- GPU 显存占用
- GPU 显存峰值
- CPU 内存占用

为什么应该补：

- 比赛主矛盾是显存压力
- 如果没有资源指标，就不能证明问题真的出在 KV 压力上

### B. vLLM 已有缓存指标

- `vllm:kv_cache_usage_perc`
- `vllm:prefix_cache_queries`
- `vllm:prefix_cache_hits`
- `vllm:num_requests_running`
- `vllm:num_requests_waiting`

为什么应该补：

- 这些都是 vLLM 已经能提供的现成指标
- 几乎不用额外改源码
- 适合当前 baseline 阶段

### C. 共享前缀场景指标

- prefix 查询次数
- prefix 命中次数
- 共享前缀场景下的 TTFT 变化

为什么应该补：

- 共享前缀不是比赛主线，但它是验证缓存命中最直接的场景
- 如果完全不测命中，缓存系统的解释力会不够

## 5.3 后续再补

这些指标不是不重要，而是应该放到你们自己的 KVCache/offload 代码接入后再测。

### A. block 行为指标

- `kv_block_alloc_total`
- `kv_block_access_total`
- `kv_block_evict_total`
- `kv_block_offload_total`
- `kv_block_reload_total`

### B. 分层命中指标

- `kv_gpu_hit_blocks_total`
- `kv_cpu_hit_blocks_total`
- `kv_miss_blocks_total`

### C. 迁移代价指标

- `offload_latency`
- `reload_latency`
- 平均每次迁移 block 数
- 迁移失败次数

### D. 迁移效果指标

- 无效淘汰率
- 回填触发率
- 回填后延迟变化

为什么后补：

- 这些指标已经不是原始 vLLM baseline 能自然提供的
- 更适合放在你们自己的 connector / worker 路径里做埋点

## 6. 当前阶段不建议强行测的指标

有些指标看起来很全面，但现在不适合硬上。

### 6.1 “推理准确率”不适合作为当前主指标

原因：

- 当前阶段主任务是系统 baseline，不是模型能力评测
- 你现在更关心吞吐、延迟、资源和命中

可以做的最小正确性检查是：

- 服务返回是否正常
- 结果是否明显错乱
- 是否出现异常崩溃

### 6.2 “Prefill 命中率 / Decode 命中率”当前不必强拆

原因：

- 这个指标理论上很好，但当前阶段未必容易稳定拿到
- 如果没有明确埋点，硬拆会增加大量实现成本

当前更现实的替代做法是：

- 先测共享前缀场景
- 先看 `prefix_cache_queries` 和 `prefix_cache_hits`
- 等后续 KVCache 版本再补更细粒度命中统计

### 6.3 “无效淘汰率”当前先不作为 baseline 核心指标

原因：

- 这是很强的方案专属指标
- 原始 vLLM baseline 阶段没有足够上下文支持精确计算

## 7. 推荐的监控来源

## 7.1 第一来源：vLLM 自带监控

当前最建议优先用来打底。

适合采：

- `kv_cache_usage`
- prefix cache 指标
- 请求运行/等待状态

优点：

- 不需要大改代码
- 和原始 vLLM baseline 天然匹配

## 7.2 第二来源：外部资源采集

适合采：

- GPU 显存
- GPU 利用率
- CPU 内存

优点：

- 不侵入推理框架
- 能直接补足“显存压力”这条比赛主线

## 7.3 第三来源：你们自定义埋点

适合采：

- block 生命周期
- offload/reload 次数
- 分层命中
- 迁移耗时

优点：

- 最贴比赛方案

缺点：

- 当前阶段实现成本更高

## 8. 当前阶段推荐测试指标表

| 指标类别 | 当前是否必须 | 具体指标 | 来源 |
|---|---|---|---|
| 可用性 | 是 | 成功率、失败数 | `vllm bench serve` / 请求日志 |
| 性能 | 是 | req/s、tok/s、TTFT、TPOT | `vllm bench serve` |
| 长度分档 | 是 | 2K、8K、16K | benchmark 配置 |
| 模型环境 | 是 | 模型名、版本、GPU、上下文长度 | 环境记录 |
| 资源 | 应补 | GPU 显存、CPU 内存 | 外部采集 |
| 缓存占用 | 应补 | `kv_cache_usage_perc` | vLLM `/metrics` |
| prefix 命中 | 应补 | `prefix_cache_queries`、`prefix_cache_hits` | vLLM `/metrics` |
| block 行为 | 后补 | alloc/access/evict/offload/reload | 自定义埋点 |
| 分层命中 | 后补 | GPU hit、CPU hit、miss | 自定义埋点 |
| 迁移代价 | 后补 | offload/reload latency | 自定义埋点 |

## 9. 当前额外采集指标及其作用说明

除了 `vllm bench serve` 默认输出的吞吐和时延指标外，当前阶段还建议额外采集以下指标。  
这些指标的作用，不是替代性能指标，而是帮助解释性能变化背后的原因。

### 9.1 `vllm:kv_cache_usage_perc`

作用：

- 观察 KV cache 占用是否随上下文长度明显升高
- 帮助判断长上下文场景中的压力是否主要来自 KV cache
- 为后续 offload 版本提供对照基线

为什么有必要：

- 比赛主线就是显存压力与 KV cache 管理
- 如果只看吞吐和延迟，不看 cache 占用，很难证明问题根源

### 9.2 `vllm:prefix_cache_queries`

作用：

- 观察共享前缀场景下，系统是否真的在执行 prefix cache 查询

为什么有必要：

- 如果有共享前缀测试，但查询次数几乎不变，说明命中链路可能根本没有被触发

### 9.3 `vllm:prefix_cache_hits`

作用：

- 观察共享前缀场景下，是否真正发生 prefix cache 命中

为什么有必要：

- 共享前缀命中组的核心目的，就是验证缓存复用是否可被观测
- 没有这项指标，就只能看到“延迟变化”，但无法确认是否真的是命中带来的

### 9.4 `vllm:num_requests_running`

作用：

- 观察系统当前同时运行中的请求数
- 用于判断压力场景下系统是否持续处于高占用状态

为什么有必要：

- 在混合压力组中，这个指标可以帮助解释吞吐与资源压力的关系

### 9.5 `vllm:num_requests_waiting`

作用：

- 观察请求是否开始排队
- 用于判断系统是否因为资源紧张或调度拥塞而积压请求

为什么有必要：

- 如果只看平均延迟，很容易忽略排队问题
- 这个指标对混合压力组尤其重要

### 9.6 GPU 显存占用

作用：

- 直接观察不同场景下 GPU 显存的使用变化
- 帮助判断长上下文和超长上下文是否已经逼近显存压力边界

为什么有必要：

- 比赛最核心的问题之一就是显存压力
- 这是最直接、最有说服力的资源指标

### 9.7 CPU 内存占用

作用：

- 记录当前原始 baseline 的 CPU 内存占用水平
- 为后续本地 offload 版本做资源对照

为什么有必要：

- 后续如果 KV 从 GPU 迁到 CPU，CPU 内存会成为关键资源
- 现在先记录 baseline，后面才能判断 offload 带来的资源转移是否值得

### 9.8 系统负载（load average）

作用：

- 辅助观察机器整体压力
- 帮助区分“是 GPU 压力为主”还是“系统整体都开始吃紧”

为什么有必要：

- 混合压力组下，单看 GPU 指标可能不够
- 需要有一个系统级背景指标帮助解释异常

### 9.9 这些额外指标的总体意义

这些额外采集指标本质上是为了把：

- “系统变慢了”

进一步解释成：

- 是因为上下文变长导致 KV cache 占用升高
- 是因为共享前缀命中没有发生
- 是因为请求开始排队
- 是因为 GPU 或 CPU 资源已经接近上限

因此，当前阶段建议把这些指标视为：

- 对吞吐和时延指标的解释层
- 对比赛主线的支撑证据

而不是把它们看成可有可无的附加项。

## 10. 对应当前代码方案的建议

基于这套指标体系，当前阶段最合适的代码方案是：

### 9.1 保留 `vllm bench serve`

用于输出：

- req/s
- tok/s
- TTFT
- TPOT

### 9.2 外部补一个采集脚本

用于补：

- `/metrics`
- GPU 显存
- CPU 内存
- 结果落盘

### 9.3 后续再做比赛专属埋点

放到：

- connector
- worker
- KV manager

而不是今天就去大改 vLLM 主干。

## 11. 结论

当前阶段不是要把截图里的所有指标一次性测完，而是要围绕比赛主线，建立一套分阶段、可落地的指标体系。

最合理的做法是：

1. 先用 `vLLM` 自带指标和 benchmark 建立 baseline
2. 再用外部资源采集补齐显存和运行状态
3. 最后在你们自己的 KVCache/offload 代码里补 block 级和迁移级指标

因此，当前阶段真正应该优先测的是：

- 吞吐
- 延迟
- 显存压力
- kv cache 占用
- prefix 命中

而不是一开始就被一张图里的全部候选指标绑住。
