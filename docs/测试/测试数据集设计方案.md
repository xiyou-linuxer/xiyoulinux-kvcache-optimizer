# 测试数据集选型与设计说明

## 1. 文档目的

本文档用于说明本项目测试数据集的设计思路、选取原则、推荐来源与落地方案，为后续原始 vLLM baseline 测试、KV Cache offload 版本测试、监控指标采集和结果对比提供统一依据。

本项目不是通用模型能力评测，而是围绕 KV Cache 热度监控、分层迁移、缓存命中和长上下文推理性能展开的系统型比赛。因此，测试数据集不能只追求“常见 benchmark”，而必须服务于以下核心问题：

- 是否能稳定触发 KV Cache 的分配、命中、驱逐、卸载和回填
- 是否能覆盖短上下文、长上下文、超长上下文、共享前缀、混合压力等关键场景
- 是否能支撑吞吐、延迟、命中率、有效上下文扩展等赛题核心指标评估
- 是否能兼顾真实性、可控性、可重复性与后续扩展性

## 2. 为什么数据集选取很关键

对于这个比赛，数据集不是“顺手找几份公开问答数据”就够了，而是测试方案能否成立的前提。

原因有三点：

### 2.1 赛题评估的是系统行为，不只是模型答题能力

比赛真正关心的是：

- KV Cache 命中率
- 平均访问延迟
- 推理吞吐率
- GPU 等待迁移的开销
- 有效上下文长度扩展能力

这些指标是否能被测出来，直接取决于测试数据是否能触发对应的系统行为。  
如果数据普遍很短，就很难逼出显存压力，也看不到 offload 的收益。  
如果数据之间没有共享前缀，就很难观察 prefix reuse 和 block 复用行为。  
如果只有单一场景，就无法说明系统在真实负载下是否稳定。

### 2.2 不同数据场景对应不同验证目标

这个项目至少需要区分下面几类场景：

- 短上下文：验证基础功能、监控开销和轻负载性能
- 中长上下文：验证常规长文本场景下的吞吐和延迟变化
- 超长上下文：验证显存压力下 offload 是否真正带来上下文扩展
- 共享前缀：验证 prefix/prefill 命中和缓存复用效果
- 混合压力：验证不同长度请求并发时的稳定性和资源竞争

也就是说，单一数据集很难覆盖全部目标，必须采用“多组数据 + 分工明确”的设计。

### 2.3 这个项目既需要真实数据，也需要可控数据

真实开源数据集的作用是增强可信度，让结果更像真实场景。  
但系统测试还需要很多“人为构造”的场景，例如：

- 大量请求共享同一长前缀
- 固定上下文长度阶梯增长
- 长短请求按指定比例混合
- 异常输入或超长输入边界测试

这些场景通常无法只靠公开数据集自然得到，因此必须结合合成数据。

## 3. 数据集设计原则

本项目的数据集设计遵循以下原则。

### 3.1 以系统评测目标为中心，而不是以榜单为中心

数据集选取优先服务于系统行为验证，而不是追求“跑了很多热门 benchmark”。  
每一组数据都必须能回答一个明确问题，例如：

- 这组数据是否能测共享前缀命中
- 这组数据是否能逼出显存压力
- 这组数据是否能稳定观察回填行为

### 3.2 真实性与可控性结合

- 开源真实数据：保证实验有代表性
- 合成数据：保证长度、前缀结构、并发模式可控

两者结合后，既能做对外展示，也能做内部调试和回归测试。

### 3.3 数据集与指标一一对应

不同数据组应明确绑定核心指标：

- 共享前缀组重点看 `prefix_cache_hits`、缓存命中率、重复计算减少情况
- 超长上下文组重点看 GPU 显存占用、CPU 内存占用、吞吐下降幅度、是否支持更长上下文
- 混合压力组重点看请求排队、吞吐稳定性、P99 延迟、资源竞争

### 3.4 baseline 与优化版必须使用同一批数据

为了保证结论可比较，同一组样本必须分别在以下模式下运行：

- 原始 vLLM baseline
- 开启监控的版本
- 开启 KVCache offload 的版本

否则结果无法证明优化是否有效。

### 3.5 第一阶段聚焦单机场景，但预留扩展空间

当前阶段重点是本地 GPU-CPU offload，因此数据集优先围绕单机长上下文与 prefix reuse 场景设计。  
后续如果进入 SSD、远端节点或结构感知策略阶段，可在当前分组基础上扩展，而不需要推翻重来。

### 3.6 五组场景 baseline 是主线，随机 benchmark 只是补充

本项目最终要交付的 baseline 主体，是这五组比赛场景数据。  
固定长度的随机 benchmark 可以保留，但它的定位应该是：

- 补充性的纯性能参考线
- 用于控制变量和观察长度退化趋势

而不应替代真实场景数据组本身。

## 4. 数据集总体设计

建议采用“开源真实数据 + 合成控制数据”的双轨方案，并按五组组织。

| 分组 | 主要目标 | 是否必须 | 推荐来源 |
|---|---|---|---|
| 短上下文基础组 | 验证基础功能与监控开销 | 是 | 合成数据 + 少量真实问答 |
| 中长上下文主测试组 | 验证常规长文本场景性能 | 是 | LongBench 子集 |
| 超长上下文极限组 | 验证上下文扩展与显存压力 | 是 | LongBench v2 / NeedleBench / 合成长文本 |
| 共享前缀命中组 | 验证 prefix/prefill 命中与 block 复用 | 是 | 自合成数据为主 |
| 混合压力组 | 验证并发稳定性与资源竞争 | 是 | 从前四组按比例混合 |

## 5. 各数据组设计说明

### 5.1 短上下文基础组

#### 目标

- 验证原始 vLLM 推理流程是否正常
- 验证监控采集链路是否打通
- 观察轻负载下开启监控或 offload 的额外开销

#### 长度建议

- 输入长度：512 到 2K token
- 输出长度：64 到 256 token

#### 数据类型建议

- 短问答
- 短摘要
- 简单改写
- 单轮对话

#### 推荐来源

- 少量通用问答样本
- 自构造短 prompt

#### 主要观察指标

- 服务是否稳定返回
- tokens/s
- TTFT
- 平均请求延迟
- 监控是否成功采集

#### 价值

这组数据不是为了体现 offload 收益，而是为了给整个测试流程打底，避免一开始就陷入长上下文复杂问题。

### 5.2 中长上下文主测试组

#### 目标

- 验证常规长文本任务下 offload 对性能与资源的影响
- 观察长上下文场景中的缓存占用和整体吞吐变化

#### 长度建议

- 输入长度：4K 到 8K token
- 输出长度：128 到 512 token

#### 推荐来源

- LongBench 中偏长的 QA / 多文档 QA 子集

#### 推荐任务类型

- 单文档问答
- 多文档问答
- 长文检索问答

当前阶段不建议优先把摘要类任务作为主组，原因是：

- 问答/检索类任务更容易形成稳定的长上下文压力
- 更适合观察服务在长文本查找和定位场景下的延迟变化

#### 主要观察指标

- tokens/s
- 平均延迟
- TTFT
- GPU 显存占用
- CPU 内存占用
- KV cache usage

#### 价值

这组数据最接近“正常长上下文业务场景”，应作为第一阶段最主要的对比组。

### 5.3 超长上下文极限组

#### 目标

- 逼近或超过 baseline 的显存压力边界
- 验证启用 offload 后是否能支持更长上下文
- 观察长序列下的吞吐退化与迁移开销

#### 长度建议

- 输入长度：8K 到 32K token
- 输出长度：128 到 512 token

#### 推荐来源

- NeedleBench 的 retrieval 风格子集
- LongBench 中更长的样本
- 合成长文档或拼接长代码片段

#### 主要观察指标

- 能否成功运行到目标上下文长度
- GPU 显存占用峰值
- CPU 内存占用变化
- 平均延迟与 P99 延迟
- 吞吐下降幅度
- 是否发生明显卸载/回填

#### 价值

这一组直接对应比赛中的“支持更长序列长度”和“减少 GPU 等待迁移时间”这两个关键卖点，是非常重要的展示数据。

这里优先选 retrieval 风格数据，而不是一开始就选最复杂的 reasoning 数据，原因是：

- 当前重点是系统压力与可观测性
- retrieval 风格更容易稳定制造超长 haystack
- 更适合先看显存、延迟和稳定性

### 5.4 共享前缀命中组

#### 目标

- 验证 prefix/prefill 命中是否真实发生
- 验证共享前缀下是否减少重复计算和重复搬运
- 观察 block 复用、命中率和回填行为

#### 数据构造方式

建议以自合成为主，不依赖公开数据集自然形成。

构造方式示例：

- 固定一段长 system prompt 或长文档前缀
- 让多个请求共享这段前缀
- 每个请求只在末尾拼接不同问题

例如：

- 同一篇长文档 + 不同提问
- 同一段代码仓库说明 + 不同检索问题
- 同一系统提示词 + 不同用户指令

#### 长度建议

- 共享前缀长度：2K 到 8K token
- 后缀问题长度：64 到 512 token

#### 主要观察指标

- `vllm:prefix_cache_queries`
- `vllm:prefix_cache_hits`
- 命中率
- 重复请求的 TTFT 变化
- block 复用次数
- reload 是否减少

#### 价值

这组数据是最能体现“KV 复用”和“缓存命中”价值的场景。  
如果缺少这一组，测试方案对缓存系统的解释力会明显不足。

### 5.5 混合压力组

#### 目标

- 验证短请求和长请求并发混跑时的稳定性
- 观察资源竞争、排队和尾延迟
- 为后续调度策略或阈值优化提供依据

#### 设计方式

从前四组中按比例抽样，形成混合请求流，例如：

- 50% 中长上下文
- 20% 超长上下文
- 20% 共享前缀请求
- 10% 短请求

并设置不同并发档位：

- 2
- 4
- 8
- 16

#### 主要观察指标

- 吞吐
- P50/P99 延迟
- 运行中请求数
- GPU/CPU 资源占用波动
- 是否频繁触发 offload/reload

#### 价值

真实系统负载通常不是“所有请求都一样长”，因此这组数据能补足前面几组偏理想化的问题。

## 6. 推荐开源数据集及选用理由

### 6.1 LongBench

#### 适合原因

- 是长上下文测试中较常见的公开基准
- 覆盖单文档 QA、多文档 QA、长对话、代码等多种任务
- 样本类型丰富，便于切分出中长上下文和超长上下文子集

#### 在本项目中的作用

- 作为中长上下文主测试组的主要真实数据来源
- 为超长上下文组提供部分真实样本
- 更适合作为“常规真实长文本场景”的代表

#### 注意点

- 不需要全量使用，应优先抽取与长输入相关的子任务
- 重点是上下文长度和系统行为，不是追求完整复现原 benchmark 排名
- 当前阶段更建议优先选长文 QA / 多文档 QA 子集，而不是随便挑一个任务

### 6.2 NeedleBench

#### 适合原因

- 专门针对长文本中的信息检索和推理场景
- 很适合验证长上下文下模型是否还能正确定位关键信息
- 更容易构造“长度很长但关键点很少”的检索型场景

#### 在本项目中的作用

- 作为超长上下文极限组的重要补充
- 用于验证长上下文下系统是否还能稳定支持推理
- 当前阶段更适合作为“超长上下文压力数据”，而不是模型 reasoning 排名数据

#### 注意点

- 它更偏长文检索，不足以覆盖 prefix reuse 场景
- 因此应与共享前缀合成组搭配使用
- 当前阶段优先 retrieval 风格子集，后续再补更复杂的 reasoning 子集

### 6.3 QuAC

#### 适合原因

- 是多轮问答数据
- 适合体现历史上下文逐轮累积带来的 KV 增长

#### 在本项目中的作用

- 可作为中长上下文主测试组中的多轮对话样本来源
- 用于观察上下文历史逐轮变长时的系统表现

#### 注意点

- 单条样本未必天然足够长，必要时需要拼接或筛选

### 6.4 RepoQA

#### 适合原因

- 代表代码类长上下文理解场景
- 代码文件、仓库说明、跨文件引用等内容通常上下文较长
- 对比赛展示较有说服力，因为代码场景往往更容易产生长上下文和重复前缀

#### 在本项目中的作用

- 作为代码场景补充组
- 可与共享前缀构造结合，形成“同仓库前缀 + 不同问题”测试

#### 注意点

- 如果第一阶段时间紧，可以先作为补充组而非主组

## 7. 候选数据集对比与选型结论

为了避免“只列很多名字但不知道最后用什么”，当前阶段需要把候选数据集和实际选型分开说明。

### 7.1 候选数据集对比

| 数据集 | 适合场景 | 优点 | 局限 | 第一阶段建议 |
|---|---|---|---|---|
| LongBench | 中长上下文、部分超长上下文 | 任务丰富、长文本场景多、公开认可度高 | 并非所有子任务都足够长 | 必选 |
| NeedleBench | 超长上下文、长文检索 | 很适合压长上下文和检索定位 | 不适合共享前缀复用测试 | 必选 |
| QuAC | 多轮对话 | 适合历史上下文逐轮累积 | 单样本长度可能不够长 | 可选 |
| RepoQA | 代码长上下文 | 代码场景展示力强 | 准备和筛选成本偏高 | 可选 |
| 自合成共享前缀数据 | 共享前缀命中 | 可控、稳定、最适合测命中 | 真实性不如公开数据 | 必选 |
| 自合成短 prompt 数据 | 基础功能、轻负载 | 成本低、最适合打通流程 | 不能体现长上下文优势 | 必选 |

### 7.2 第一阶段实际选型结论

结合时间成本、当前任务目标和比赛重点，第一阶段建议实际采用以下组合：

- 短上下文基础组：自合成短 prompt
- 中长上下文主测试组：LongBench 的长文 QA / 多文档 QA 子集
- 超长上下文极限组：NeedleBench 的 retrieval 风格子集
- 共享前缀命中组：自合成共享前缀数据

这一版已经足够覆盖：

- 基础可用性
- 常规长上下文性能
- 超长上下文显存压力
- prefix reuse 命中

### 7.3 第一阶段暂不优先纳入的候选

以下数据集不是不能用，而是当前阶段优先级稍低：

- QuAC
  - 原因：更偏多轮对话补充组，不是今天 baseline 的主线
- RepoQA
  - 原因：适合后续代码场景展示，但第一阶段先保证主测试链路跑通更重要

也就是说，当前不是“数据集越多越好”，而是先让每一组都有明确职责。

## 8. 为什么共享前缀组必须自己构造

共享前缀组不建议完全依赖开源数据集自动生成，原因如下：

- 公开数据集的前缀重复结构通常不稳定
- 很难保证多个请求共享完全一致且足够长的 prefix
- 无法精确控制 prefix 长度、后缀长度和重复次数
- 不利于和 `prefix_cache_hits` 等指标精确对应

因此建议手工或脚本合成以下模式：

- 10 到 50 个请求共享同一前缀
- 前缀长度分档为 2K、4K、8K
- 每个请求只在结尾替换一个问题

这样做的好处是：

- 命中行为更稳定
- 结果更容易解释
- 可重复性更强
- 更适合做回归测试

## 9. 数据集与比赛指标的映射关系

| 比赛关注指标 | 最适合的数据组 | 说明 |
|---|---|---|
| KV 缓存命中率 | 共享前缀组、中长上下文组 | 共享前缀组最直接，中长上下文组做补充 |
| 平均访问延迟 | 中长上下文组、混合压力组 | 反映常规场景和并发场景下的变化 |
| 吞吐率提升幅度 | 中长上下文组、混合压力组 | 最适合做 baseline 对比 |
| 有效上下文长度扩展倍数 | 超长上下文组 | 直接观察 baseline 能跑多长，offload 能跑多长 |
| GPU 等待迁移开销 | 超长上下文组、混合压力组 | 压力大时更容易暴露问题 |
| 缓存复用效果 | 共享前缀组 | 最能体现 prefix/prefill 价值 |

## 10. 第一阶段推荐落地方案

如果当前时间有限，建议第一阶段不要铺太大，优先使用以下最小可落地组合：

### 10.1 必选组

- 短上下文基础组
- 中长上下文主测试组
- 超长上下文极限组
- 共享前缀命中组

### 10.2 可选补充组

- 混合压力组
- RepoQA 代码场景组

### 10.3 推荐数据来源组合

- 短上下文基础组：自构造短 prompt
- 中长上下文主测试组：LongBench 子集
- 超长上下文极限组：LongBench 长样本 + NeedleBench
- 共享前缀命中组：自合成

这个组合的优点是：

- 成本可控
- 覆盖面够用
- 能直接对应监控和测试指标
- 适合先做 baseline，再做 offload 对比

### 10.4 第一阶段建议样本规模

为了今天就能开始跑，并且结果可控，建议先使用小规模样本：

| 数据组 | 建议样本数 | 用途 |
|---|---|---|
| 短上下文基础组 | 10 到 20 | 检查服务和监控链路 |
| 中长上下文主测试组 | 10 到 20 | 形成第一版真实 baseline |
| 超长上下文极限组 | 5 到 10 | 观察显存压力和长上下文退化 |
| 共享前缀命中组 | 10 到 20 | 看 prefix 查询和命中变化 |

如果今天时间非常紧，可以先压缩为：

- 短上下文组 10 条
- 共享前缀组 10 条
- LongBench 长文 QA 子集 10 条

这样就已经能产出第一版 base 数据。

## 11. 数据集筛选与落地注意事项

### 11.1 不要一上来追求全量 benchmark

第一阶段更适合做“代表性子集”，而不是跑完整公开基准。  
原因是当前重点是系统验证与监控闭环，而不是刷模型榜单。

### 11.2 要先统一 token 长度分桶

建议所有样本先按 token 长度分桶，例如：

- 0.5K 到 2K
- 2K 到 8K
- 8K 到 16K
- 16K 到 32K

这样后续测试更容易比较，也便于解释结果。

### 11.3 每组样本数量不用太大，但要稳定

第一阶段建议每组先选：

- 20 到 50 条代表性样本

重点是：

- 长度分布清楚
- 场景定义清楚
- 可重复运行

而不是样本数越多越好。

### 11.4 数据集文档里要写清楚“为什么选”

文档中不能只写数据集名字，还要明确：

- 这个数据集覆盖什么场景
- 对应验证哪个比赛指标
- 为什么比别的数据集更适合当前阶段

这会让你的文档从“资料整理”变成“测试设计”。

## 12. 结论

本项目的数据集设计应服务于 KV Cache 系统验证，而不是简单复用通用问答 benchmark。  
因此，最合理的方案不是只选一个公开数据集，而是采用“真实数据 + 合成数据”的组合，并围绕短上下文、中长上下文、超长上下文、共享前缀和混合压力五类场景建立测试集。

在第一阶段，建议优先使用：

- LongBench 的长文 QA / 多文档 QA 子集承担常规长上下文测试
- NeedleBench 的 retrieval 风格子集承担超长上下文压力测试
- 自合成共享前缀数据承担命中与复用测试
- 自构造短 prompt 承担基础验证

这样既能贴合比赛对命中率、延迟、吞吐和上下文扩展的要求，也能为后续监控指标采集、baseline 对比和 KVCache 优化分析提供稳定基础。
