## Slide 3

项目背景

随着上下文长度增长，KV Cache 正在成为 LLM 推理中最突出的显存占用来源。

问题来源

显存压力的直观变化

- • LLM 推理需要保存历史 token 的 Key-Value 缓存，以减少重复 Attention 计算。
- • 当上下文从 2K、8K 增长到 16K、32K 甚至更高时，KV Cache 会近似线性膨胀。
- • 在资源受限设备上，KV Cache 会逐渐挤占模型权重、激活与运行时调度所需要的显存预算。

## Slide 14

创新点与增量贡献（1/2）

- 本项目选择性参考了部分开源项目FlexKV 的总体思路，但并非整体照搬；
- 本页展示与版本适配、任务管理和布局解耦相关的代表性增量贡献。
- 在项目开发期间，我队积极与flexkv开发者沟通，并累计形成 8 项公开开发记录，其中包括 6 项已合并 PR、1 项待合并 PR 和 1 项 Issue 讨论，体现了我队对项目底层机制的持续投入。

vLLM 0.23 版本布局适配

批处理任务生命周期重构

1

2

- • 面向非 MLA 场景新增 `LAYERBLOCK` 路径，适配 GPU KV Cache 新布局
- • 用 stride / pitch 描述布局差异，避免将旧版布局假设写死在实现中

- • 下层与上层共同跟踪 merged task，修正“子任务视角等待”错配
- • 降低状态丢失、任务泄漏与完成通知错位风险

同步 GET 完成清理机制

3

- • 为 `complete=false` 任务增加标志位，区分“已被上层查询”状态
- • 真正完成时可立即清理字典项，而非长期滞留等待超时

CPU / SSD 异构 layout 解耦

4

- • SSD 固定 `BLOCKFIRST`，CPU 保持上层访问友好的布局
- • 借助 `preadv/pwritev` 连接离散内存与连续磁盘区域
